# AAPLCAD SDK 详细开发规划安排（Apple-first 路线）

## 1. 规划目标

本规划基于《AAPLCAD SDK 设计文档（Apple-first / Apple Silicon 原生）》展开，目标是将 AAPLCAD SDK 从概念设计推进为一个真正 Apple-first 的现代 CAD 技术平台。它不仅运行在 macOS 和 Apple Silicon 上，而且从渲染、拾取、输入、交互、跨设备协同和开发者开放性上都围绕 Apple 生态重新设计，以形成与传统 CAD SDK 明确不同的竞争力。

该规划重点解决以下问题：

- 如何分阶段推进，避免一开始陷入“大而全”风险
- 如何组织核心模块开发顺序，形成最小可验证闭环
- 如何安排团队角色与产出边界
- 如何设定阶段验收标准、性能目标和风险控制点
- 如何在保证架构正确性的同时尽快形成可运行成果
- 如何把 Apple-first 差异化能力纳入主线，而不是后期附加项

---

## 1.1 本项目的战略方向

本项目要刻意避免落入“传统 CAD for Mac”路线，而是瞄准以下方向：

- 完全 Apple-first
- Metal-first 渲染与拾取
- 为触控板、键盘鼠标、多窗口、多显示器和 macOS 原生生产力工作流而设计
- 几何层、显示层、交互层一起重构
- 不背传统 CAD SDK 的历史包袱、许可证和黑盒限制

因此，开发规划必须保证：

- macOS 原生交互能力从 Phase 0 就进入架构设计
- 渲染与拾取统一按 Metal-first 推进
- 输入抽象、桌面工作流、多窗口协同不是“增强功能”，而是路线组成部分
- 对传统文件格式和兼容需求保持理性，但不让其主导底层架构

---

## 2. 总体开发策略

### 2.1 开发原则

整个项目采用“Apple-first 架构先行、闭环优先、逐层递进”的策略。

原则如下：

1. 先建立稳定骨架，再堆叠复杂能力
2. 先完成 2D 专业 CAD 基础闭环，再进入高级 3D / DWG / 参数化
3. 先把 macOS 原生体验和 M 系列性能优势打扎实，再考虑跨平台扩展
4. 先实现 SDK 内核，再逐步扩展到上层专业应用
5. 每个阶段都必须有可展示、可测试、可回归的成果物
6. 任何与 Apple 生态差异化直接相关的能力都不能被无限后置

### 2.2 方法论

建议采用“技术预研 + 阶段交付 + 双周迭代”的模式。

- 长期按阶段推进（Phase）
- 中期按季度里程碑推进（Milestone）
- 短期按 2 周 Sprint 执行

推荐组织节奏：

- 每 2 周一次迭代计划与评审
- 每 4 周一次里程碑检查
- 每 8~12 周进行阶段验收

---

## 3. 项目阶段划分

建议将整个 SDK 开发拆分为 7 个阶段。

## Phase 0：立项与 Apple-first 预研（约 4~6 周）

### 目标

明确技术路线、建立工程基线、验证高风险点。

### 重点任务

- 确定语言、编译链、CMake 结构、代码规范
- 预研 Apple Silicon 下 Metal 2D/3D 渲染路径
- 预研 Metal picking / overlay / transient feedback 架构
- 预研几何内核基础结构与数值容差策略
- 预研数据库对象模型（ObjectId / Transaction / Undo）
- 预研触控板、键盘、鼠标的统一事件抽象
- 预研多窗口、多显示器、命令面板和文档工作流模型
- 评估 DXF、DWG、STEP、IGES 的接入路线
- 建立性能 profiling 流程
- 确定测试体系和代码质量门禁

### 核心产出

- 技术选型说明
- 工程目录结构草案
- 第一版架构图
- Metal Viewer 原型
- macOS Input Abstraction 原型
- Geometry/Database 原型
- 风险列表与优先级矩阵

### 验收标准

- 项目基础结构已确定
- 已能运行最小 Metal 原型程序
- 已验证至少一条 macOS-first 输入抽象链路
- 已有基础对象模型原型
- 已明确高风险模块的技术方案或备选方案

---

## Phase 1：基础骨架搭建（约 6~8 周）

### 目标

建立 SDK 主体工程与基础模块，形成后续开发主干。

### 重点任务

#### 1. 工程化建设

- 建立 CMake 主工程
- 建立 `include/`、`src/`、`tests/`、`examples/` 目录体系
- 建立静态库/动态库产物规划
- 建立编译选项、警告级别、Sanitizer 策略
- 集成单元测试框架和 benchmark 框架

#### 2. Core 模块

- 数学基础类型：向量、矩阵、AABB、变换
- 错误与诊断系统
- 日志系统
- ObjectId / Handle / UUID
- 基础容器与内存分配器接口

#### 3. Platform 模块

- 线程封装
- 文件系统封装
- 时间与 profiling 基础接口
- Metal 设备与上下文封装
- 输入设备抽象接口
- macOS 前端桥接与桌面文档框架预留

#### 4. SDK 接口边界

- 公共头文件与内部实现隔离
- 基础 ABI 策略
- 命名空间与模块边界确定

### 核心产出

- 可编译 SDK 工程主干
- 基础 Core / Platform 模块
- 单元测试跑通
- 一个最小示例程序成功链接 SDK
- 最小 macOS 输入抽象接口跑通

### 验收标准

- 工程在 macOS Apple Silicon 环境稳定编译
- 核心模块有基础测试覆盖
- SDK 输出物结构稳定
- 为后续 geometry/database/graphics 开发提供明确边界
- 已为 macOS 桌面工作流、窗口系统和输入设备留好接入口

---

## Phase 2：2D 几何与数据库闭环（约 8~12 周）

### 目标

形成 2D 专业 CAD 最小可用内核。

### 重点任务

#### 1. Geometry 2D

- `Point2D`
- `Line2D`
- `Circle2D`
- `Arc2D`
- `Ellipse2D`
- `Polyline2D`
- `Spline2D`（可先做基础版）
- 测量、求交、投影、包围盒、偏移基础能力

#### 2. Database

- `Document`
- `Database`
- `Entity`
- `Layer`
- `BlockDefinition`
- `BlockReference`
- `TextStyle`
- `DimStyle`（先保留最小结构）
- 事务系统
- Undo / Redo journal

#### 3. 基础序列化

- AAPLCAD 自有文档格式初版
- 对象保存 / 加载
- 版本字段设计

### 核心产出

- 2D 图元内核
- 图元数据库管理能力
- 增删改查事务能力
- 文档保存与读取能力

### 验收标准

- 能创建、编辑、保存、读取基本 2D 文档
- 事务与撤销重做可用
- 关键几何算法有测试覆盖
- 数据结构足以支撑图形显示层接入

---

## Phase 3：Metal 图形引擎与交互系统（约 10~14 周）

### 目标

建立 macOS 原生高性能 CAD 视图系统。

### 重点任务

#### 1. Graphics Engine

- Metal 渲染上下文
- 2D Draw List
- 基础图元 tessellation/cache
- Retina 显示支持
- 视图变换
- 高亮绘制
- 选择拾取 pass
- 瞬态预览 overlay pass
- 文本与标注占位渲染方案

#### 2. Interaction Engine

- 缩放、平移、框选
- 单选、多选
- 对象捕捉初版
- 动态预览图形
- Transient graphics
- 命令上下文基础
- 触控板交互映射
- 键盘命令流与多窗口交互事件管线占位实现

#### 3. 性能优化

- 增量刷新
- 缓存策略
- GPU buffer 生命周期管理
- 大图纸视图压力测试

### 核心产出

- 可交互 CAD Viewer
- 2D 基础渲染链路
- 选择与捕捉基础能力
- 交互性能 profiling 报告
- Trackpad-first / Desktop-first 导航实验结果

### 验收标准

- 能流畅显示中等规模 2D 图纸
- 缩放、平移、选择体验稳定
- 图层显隐与对象高亮可用
- 性能达到预期基线
- 触控板交互体验达到一等支持水平

---

## Phase 4：DXF 与工程级 2D 能力完善（约 8~12 周）

### 目标

使 SDK 具备实用级 2D CAD 工程能力。

### 重点任务

- DXF 导入器
- DXF 导出器
- 图层、块、文字、标注增强
- 线型/线宽/颜色体系完善
- 打印预览基础
- 模型空间/布局空间预留
- 更多选择过滤器与编辑命令
- 工程资源管理（模板、样式、线型）
- macOS 桌面文档协同与命令面板增强原型

### 核心产出

- 与外部 CAD 文件的基本互操作
- 更接近专业 CAD 的对象体系
- 2D 工程图纸基础工作流

### 验收标准

- 能稳定导入和导出常见 DXF 文件
- 图元属性、图层、块基本正确保留
- 常见 2D 图纸可完成浏览与基础编辑

---

## Phase 5：约束与参数化（约 10~16 周）

### 目标

提升设计能力，从绘图进入“可驱动设计”阶段。

### 重点任务

- 约束图模型
- 基础几何约束：平行、垂直、同心、相切、重合
- 尺寸约束
- 参数表与表达式引擎
- 增量求解
- 过约束/欠约束诊断
- 约束 UI 与编辑反馈
- 键盘/触控板驱动草图编辑与约束推断联动

### 核心产出

- 2D 草图参数化基础能力
- 解算器与数据库联动
- 约束诊断机制

### 验收标准

- 常见 2D 草图约束可稳定求解
- 拖拽与参数修改能实时反馈
- 约束冲突能给出可理解诊断

---

## Phase 6：3D 基础与高级格式（约 16~24 周）

### 目标

让 SDK 从 2D 核心演进到完整工程 CAD 平台。

### 重点任务

- 3D 基础几何
- Mesh / Surface 基础能力
- B-Rep 预研与最小实现
- 拉伸、旋转、扫掠等初步特征
- STEP / IGES 文件接入
- 3D 显示模式：线框、着色、边线
- 3D 选择、视图操作、剖切预留
- macOS 大型工程模型审阅与批注增强
- 面向桌面协作的会话与文档状态管理增强

### 核心产出

- 可视化 3D 基础模型
- 与通用工程格式互通基础
- 进入 3D CAD 能力建设轨道

### 验收标准

- 能显示并编辑基础 3D 几何对象
- 支持部分通用工程格式导入导出
- 3D 渲染与数据结构不破坏 2D 主流程

---

## Phase 7：产品化、生态与设备协同（持续阶段）

### 目标

让 SDK 成为可交付、可扩展、可商业化的平台。

### 重点任务

- 插件系统
- 脚本接口
- 稳定 API 版本管理
- 崩溃恢复和诊断上报
- 文档体系与示例工程
- 商业授权与部署方案
- DWG 商业兼容方案落地
- macOS 平台接入文档
- 输入能力矩阵与桌面设备支持策略

### 核心产出

- 可供内部/外部团队使用的 SDK
- 插件开发能力
- 面向专业产品的运维与交付能力

### 验收标准

- SDK 文档齐全
- 示例可覆盖主要功能
- 插件接口稳定
- 商业发行路径清晰

## 4. 建议时间安排（按 18~24 个月规划）

以下为较现实的首版规划，适合做成 1.0 技术平台：

- 第 1~2 个月：Phase 0
- 第 3~4 个月：Phase 1
- 第 5~7 个月：Phase 2
- 第 8~10 个月：Phase 3
- 第 11~13 个月：Phase 4
- 第 14~17 个月：Phase 5
- 第 18~24 个月：Phase 6 + Phase 7 初步建设

如果团队规模较小（1~3 人），建议调整为：

- 先聚焦 Phase 0~3，约 9~12 个月打通完整 2D 闭环
- 再决定是否投入约束、3D、更多格式兼容和商业化生态

## 5. 团队角色建议

如果希望高质量推进，建议至少具备以下角色：

## 5.1 核心团队（最小配置 4~6 人）

### 1. 技术负责人 / 架构负责人

职责：

- 统筹总体架构
- 决策技术方向
- 控制模块边界
- 审核高风险实现

### 2. 几何内核工程师

职责：

- 负责 geometry 模块
- 求交、偏移、投影、数值稳健性
- 参与约束求解基础设计

### 3. 图形与性能工程师

职责：

- 负责 Metal 图形引擎
- 选择与渲染性能优化
- 实现缓存、增量刷新和 profiling

### 4. 交互与设备体验工程师

职责：

- 负责触控板、键盘、鼠标、多窗口输入语义抽象
- 负责命令系统与多设备交互一致性
- 定义现代 macOS-first 的 CAD 交互模式

### 5. 数据库 / 应用框架工程师

职责：

- 负责 document/database/transaction/undo
- 命令系统与文档生命周期管理
- 文件格式接入框架

### 6. macOS 原生客户端工程师

职责：

- 负责 AppKit / Objective-C++ / Swift 集成
- 窗口、输入、菜单、文档交互
- 调用 SDK 构建演示应用

### 7. QA / 自动化测试工程师

职责：

- 设计几何回归集
- 构建渲染快照测试
- 建立性能回归与发布验证流程

## 5.2 扩展角色（中期建议增加）

- 参数化/求解器工程师
- 3D 内核工程师
- 文件格式兼容工程师
- 技术文档工程师
- 产品经理 / CAD 领域专家

## 6. Sprint 执行方式建议

建议每个 Sprint 固定包含以下内容：

### 6.1 Sprint 输入

- 明确目标模块
- 明确本次功能边界
- 明确性能和测试要求
- 明确依赖阻塞项

### 6.2 Sprint 输出

- 功能代码
- 单元测试 / 集成测试
- Demo 或示例
- 已知问题列表
- 性能对比记录

### 6.3 Sprint 评审项

每个迭代必须检查：

- 功能是否可演示
- API 是否清晰
- 是否破坏现有架构边界
- 是否引入性能退化
- 是否有自动化回归

## 9. 质量保障机制

## 9.1 测试体系

建议建立 4 层测试：

1. 单元测试
   - 数学类型
   - 几何算法
   - 数据库对象行为

2. 集成测试
   - 文档加载与保存
   - 事务回滚
   - DXF 导入导出
   - 渲染数据构建

3. 回归测试
   - 几何边界案例集
   - 历史 bug case
   - 渲染快照

4. 性能测试
   - 大图纸加载
   - 高频缩放平移
   - 批量选择
   - Regen 时间

## 9.2 质量门禁

建议至少设置以下门禁：

- 新模块无测试不得合并
- 公共 API 修改必须附带文档说明
- 性能退化超过阈值需评审
- 崩溃类问题必须有复现样例
- 重要几何 bug 必须加入回归集

---

## 10. 性能里程碑建议

建议从早期就设定量化目标，而不是后期补性能：

### Milestone A：原型期

- 完成 1 万级 2D 图元平滑显示
- 视图基础交互无明显卡顿

### Milestone B：2D 闭环期

- 10 万级 2D 图元可接受缩放平移
- 选择与高亮在典型图纸下响应可控
- Trackpad-first 导航优于传统鼠标兼容模式

### Milestone C：专业化期

- 中大型图纸可稳定打开
- 增量 regen 明显优于全量重建
- 内存与统一内存占用可控

### Milestone D：高级期

- 复杂工程图与基础 3D 数据均可稳定处理
- 关键高频操作达到产品级体验
- macOS 桌面工作流链路具备演示价值

---

## 11. 风险管理安排

建议从一开始建立风险登记表，并每月更新。

### 一级风险

- DWG 兼容路径不确定
- 几何稳健性不足
- 约束求解器复杂度过高
- 大图纸性能不达标
- 图形精度与 Metal 实现冲突
- 多设备交互抽象过早耦合业务逻辑
- 非 macOS 平台探索投入过深而偏离主线价值

### 二级风险

- 文本与字体渲染复杂度超预期
- 打印与布局系统周期延长
- 代码边界失控导致后期重构成本高

### 风险应对机制

- 每个高风险模块必须有原型验证阶段
- 不在主线中同时引入多个高复杂能力
- 对外格式兼容采用“适配层”而不是直接污染核心数据库
- 任何高频操作均尽早建立 benchmark
- 对 Apple 生态新能力采用“核心抽象先行、设备实现递进”的策略

## 15. 结论

AAPLCAD SDK 的开发不能采用传统“先堆功能、后补架构”的方式，而应当以平台化思路推进：

- 先做能演进的底层骨架
- 再做 Apple-first 的 2D 专业 CAD 完整闭环
- 再做参数化、3D、空间计算、设备协同、生态和商业化

如果规划执行得当，第一年可以完成一个高质量的 macOS-first 2D CAD SDK 原型；第二年再进入参数化、3D、更多专业能力和产品化强化阶段，将更现实，也更有机会真正形成具有差异化的现代 CAD 技术底座。
